# v0.6.0 backend design

## Product boundary

Asobi stores a curated graph-shaped K/V model:

- entity identity and type;
- append-only observations per entity;
- upserted truths keyed by `(entity, key)`;
- directed relations keyed by `(from, to, relation_type)`;
- installed skill bodies keyed by skill entity name.

Markdown produced by `compact` is a one-way, human-readable projection of durable graph state.

## API v2

`api::v2` is the sole public storage contract. It exposes graph, keyword search, skills, logical snapshots, physical backup, maintenance, and task dispatch. `BackendCapabilities` reports behavior without exposing SQLite handles or SQL.

The shipped backend reports FTS5 keyword search, logical snapshots, physical backup, and multi-process support through SQLite WAL and locking.

## SQLite layout

rusqlite is the synchronous client for the local CLI and bundled SQLite is the embedded engine. SQLite tables, foreign keys, triggers, FTS5 tables, WAL, and immediate transactions are the supported implementation. Logical export/import is the portable handoff; physical backup/restore is SQLite-specific.

No alternate provider is shipped in 0.6. Backend variation is a future design topic, not an optional or experimental product feature.
