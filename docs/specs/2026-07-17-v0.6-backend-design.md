# v0.6.0 backend design

## Product boundary

Asobi stores a curated graph-shaped K/V model:

- entity identity and type;
- append-only observations per entity;
- upserted truths keyed by `(entity, key)`;
- directed relations keyed by `(from, to, relation_type)`;
- installed skill bodies keyed by skill entity name.

Markdown produced by `compact` is a one-way, human-readable projection of
durable graph state. It is not an embedding source and is not re-ingested by
the core backend.

## API v2

`api::v1` remains frozen. `api::v2` removes document/vector capability traits
and exposes only graph, keyword search, skills, logical snapshots,
provider-specific physical backup, and maintenance.

`BackendCapabilities` reports behavior without exposing driver details:

| field | SQLx SQLite | RocksDB experimental |
|---|---|---|
| `keyword_search_kind` | `fts5` | `indexed-token` or `none` |
| `logical_snapshots` | yes | yes |
| `physical_backup` | yes | experimental/explicitly verified |
| `multi_process` | yes, with SQLite locking/WAL verification | no claim until proven; RocksDB locks a database to one process |
| `experimental` | no | yes |

FTS5 ranking and stemming are SQLite capabilities, not requirements imposed on
RocksDB. The v2 search contract is a graph result with a truthful capability
report; backend-specific ranking quality is verified separately.

## SQLx SQLite layout

SQLx is the async client and bundled SQLite is the embedded engine. SQLite
tables, foreign keys, triggers, FTS5 tables, WAL, and immediate transactions
remain the supported default implementation. Logical export/import is the
portable handoff; physical backup/restore remains SQLite-specific.

## RocksDB layout

The experimental provider uses column families and ordered key prefixes. The
initial design is:

| column family | key shape | value |
|---|---|---|
| `entities` | `entity/<name>` | entity type and metadata |
| `observations` | `obs/<entity>/<id>` | observation body and timestamps |
| `truths` | `truth/<entity>/<key>` | current truth value |
| `relations` | `out/<from>/<type>/<to>` and `in/<to>/<type>/<from>` | relation metadata |
| `skills` | `skill/<entity>` | serialized skill record/body |
| `indexes` | `truth/<key>/<value>/<entity>` and token prefixes | secondary-index entries |

Entity names, keys, and relation segments must use length-prefixed or escaped
encoding so separators cannot collide with user input. Multi-record mutations
use one RocksDB `WriteBatch`, preserving atomicity across column families.

RocksDB supports atomic write batches, consistent iterators/snapshots, WAL, and
column families, but its database lock permits only one process to open a
database at a time. The provider therefore remains experimental and must not
be advertised as equivalent to the multi-process CLI behavior until the
verification task proves an explicit locking/coordination policy.
