# Asobi v0.4 — concurrency correctness + sandbox-safe open

**Date:** 2026-07-07 **Status:** DRAFT (for review) **Epic:** `asobi:v0.4` (make `init_db` safe under bursty multi-process use)

v0.4 targets the two failure modes agents actually hit in the wild:

1. **Lock storm** — a lead and its dispatched sub-agents fire several `asobi` commands in a burst and get `database is locked`.
2. **Sandbox open failure** — inside Codex (and similar landlock/seccomp sandboxes) `asobi` can't open the default XDG-global db at all.

Both trace back to `src/db.rs::init_db()` and the path fallback in `src/paths.rs`. The v0.2 `task-1` (WAL + `busy_timeout`) was the right direction but insufficient: it de-risked steady-state writes, not the _startup_ contention that every invocation now pays.

## Root causes (from investigation)

### RC1 — every invocation re-runs the full schema setup, so reads are writers

`init_db()` runs unconditionally for **every** command, incl. `search` / `show` / `graph` (`main.rs:404`). On each call it executes ~20 lock-taking statements (`db.rs:26–131`):

- `PRAGMA journal_mode = WAL` (`db.rs:27`) — momentarily takes an **exclusive** lock. WAL is a persistent db-header property; re-asserting it every open is pure contention with zero benefit.
- ~15 `CREATE TABLE/INDEX/TRIGGER IF NOT EXISTS` (`db.rs:35–131`) — each takes a schema lock even as a no-op on an existing db.
- The migration probe + conditional rebuild (`db.rs:49–88`).

Because reads pay this too, a burst of concurrent read+write commands all serialize through startup DDL. `busy_timeout` is only **5000ms** (`constant.rs:14`), and `SQLITE_LOCKED` (schema lock held by another connection) is **not** retried by the busy handler at all → `database is locked`. The `db.rs:22–25` comment shows this class was already half-known.

### RC2 — default db path is outside the sandbox writable root

`AsobiPaths::resolve_from` (`paths.rs:74`) falls through to the XDG-global dir `~/.local/share/asobi/` when no `asobi.toml` / `.asobi/` exists upward. Codex's sandbox only permits writes under the workspace (cwd) plus a few temp dirs, so:

- `create_dir_all` / opening the db / creating the `-wal` + `-shm` sidecars fails with `EROFS`/`EPERM` → "unable to open database file".
- Even on a writable path, **WAL** needs an mmap'd `-shm` segment; some seccomp profiles block the required syscalls → "disk I/O error".

Env overrides (`ASOBI_HOME`, `ASOBI_DATABASE_URL`) exist but are undocumented for Codex, and there is no WAL fallback nor an actionable error.

## Design decisions (for review)

1. **Schema-version gate via `PRAGMA user_version`.** Introduce `SCHEMA_VERSION` (a monotonically bumped `i64`). On open, read `user_version`; if it already equals `SCHEMA_VERSION`, **skip all DDL, the migration probe, and the WAL pragma** — set only the per-connection pragmas (`foreign_keys`, `busy_timeout`). This makes the common path — and _all_ reads — lock-free. This is the load-bearing fix; everything else is belt-and-suspenders.
2. **One-time setup runs under `BEGIN IMMEDIATE`.** When `user_version` differs, take the write lock once, run setup + existing migrations, bump `user_version`, commit. Concurrent starters wait on `busy_timeout` once, then observe the bumped version and skip. `journal_mode = WAL` moves _into_ this one-time block (it persists in the header).
3. **WAL is best-effort with a rollback-journal fallback.** If `journal_mode = WAL` errors, or `ASOBI_JOURNAL_MODE` is set, fall back to `DELETE` (no `-shm`, sandbox-safe). Steady-state single-writer + `busy_timeout` still works.
4. **No new persistent state, no config-schema change.** `user_version` is a built-in SQLite header field; nothing to migrate, and old binaries ignore it.

Order rationale: 1–3 are independent, low-risk hardening that helps even before the gate lands. 4 is the load-bearing change; 5 tightens its correctness; 6 proves it; 7 documents the sandbox story.

### task-1 · `busy_timeout` raise + env override _(reliability, ~5 lines)_

Bump the default from 5000ms; read `ASOBI_BUSY_TIMEOUT` (ms) when set. Build the pragma string at runtime instead of the `constant.rs:14` literal. **Done:** default is the new value; `ASOBI_BUSY_TIMEOUT=20000 asobi stats` uses it; `make check` green.

### task-2 · Actionable open-failure error _(sandbox UX, ~15 lines)_

Wrap the `create_dir_all` (`db.rs:12`) and `Builder::…build()` (`db.rs:17`) failures with `anyhow` context naming the **resolved db path** and a one-line hint: run `asobi init --local`, or set `ASOBI_HOME` to a sandbox-writable dir. **Done:** opening a read-only path prints the path + hint, not a bare "unable to open database file".

### task-3 · Journal-mode fallback + env override _(sandbox correctness)_

In the one-time setup (post-task-5) / current pragma loop, attempt `journal_mode = WAL`; on error fall back to `DELETE` and log at `warn`. Honor `ASOBI_JOURNAL_MODE=DELETE|WAL|TRUNCATE` as an explicit override. **Done:** on a filesystem/sandbox where WAL fails, `init_db` still opens; forcing `ASOBI_JOURNAL_MODE=DELETE` produces no `-shm`/`-wal` files.

### task-4 · `user_version` schema-version gate _(the load-bearing fix)_

Add `SCHEMA_VERSION` const. At the top of `init_db`, `PRAGMA user_version`; if it equals `SCHEMA_VERSION`, set only per-connection pragmas (`foreign_keys`, `busy_timeout`) and return early. Otherwise run the existing setup path. Keep the TEXT→INTEGER `id` migration (`db.rs:49–88`) reachable on the mismatch path so 0.1–0.3 dbs still upgrade. **Done:** a warm db runs **zero** CREATE/`ALTER` statements on open (verify via a statement-count test or tracing); a fresh db and a legacy db both come up correctly.

### task-5 · One-time setup under `BEGIN IMMEDIATE` _(correctness of the gate)_

Wrap the mismatch-path setup in an immediate transaction; bump `user_version` inside it; move `journal_mode = WAL` here (persists in header, needn't re-run). Re-check `user_version` after acquiring the lock to avoid a double-migrate race. **Done:** two processes started simultaneously against a cold db → exactly one migrates, neither errors, schema is correct.

### task-6 · Concurrency regression test _(proves RC1 closed)_

Spawn N (≥8) processes/tasks doing interleaved `obs`/`truth`/`search` against one db; assert zero non-zero exits and no `database is locked`. Prefer an integration test under `tests/` driving the built binary (real multi-process), not in-process connections. **Done:** the test is red on `main`, green after task-4/5; `make check` green.

### task-7 · Codex / sandbox docs + doctor hint _(discoverability)_

Add a "Running under a sandbox (Codex, etc.)" section to `README.md` + `docs/usage.md`: prefer `asobi init --local` (puts `.asobi/` under cwd, inside the sandbox) or `ASOBI_HOME=$PWD/.asobi`; note `ASOBI_JOURNAL_MODE=DELETE` for WAL-hostile filesystems; include a Codex `writable_roots` snippet for keeping the global graph. Optionally surface the resolved path + journal mode in `asobi stats`. **Done:** a fresh reader can get asobi working inside Codex from the docs alone.

## Non-goals / deferred

- **No server / daemon / connection-pool.** The one-process-per-command model stays; the gate is enough to make it safe.
- **No cross-process advisory locking** beyond SQLite's own (busy_timeout + IMMEDIATE). Revisit only if task-6 still flakes.
- **`asobi doctor`** as a standalone diagnostics command — fold the minimal hint into task-7's `stats` output for now; promote to its own command in a later release if it earns its keep.

## Acceptance for the 0.4.0 cut

- Warm-open path runs no schema DDL (task-4 test).
- N-parallel-writer test green (task-6).
- `init_db` opens on a WAL-hostile / read-only-XDG sandbox given a writable `ASOBI_HOME` or `--local` (task-2/3), with an actionable error otherwise.
- `make check` green; CHANGELOG `v0.4.0` entry written.
