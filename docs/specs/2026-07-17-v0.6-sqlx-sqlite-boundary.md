# v0.6.0 SQLx + SQLite product boundary

## Decision

Asobi 0.6.0 keeps the local graph workflow and replaces the libSQL default provider with embedded SQLite accessed through SQLx. The product contract is the graph, FTS5 search, skills, task dispatching, backup/restore, and the one-way `compact` projection from graph state to Markdown topics.

The Markdown topics written by `compact` remain a human-readable, reviewable projection and archival surface. They are not automatically re-ingested into an embedding or vector-search tier.

## Removed from the core contract

- `ingest` and semantic document `query`;
- fastembed model downloads and embedding-backed skill recall;
- vector chunks, vector indexes, and duplicate-topic detection based on vectors;
- the `DocumentStore` and `DocumentMaintenanceStore` capability requirements for the core storage backend.

The old document/vector implementation may be removed as part of the migration or preserved separately as an experimental follow-up. It must not remain a required dependency, startup path, or release verification gate for the core CLI.

## Retained behavior

- Graph entities, observations, truths, relations, and FTS5 search;
- `skills` installation, listing, update, removal, and body display without embedding a skill body;
- `tasks plan|list|dispatch|sync|close` and their response schemas;
- `compact`, which writes durable graph knowledge to Markdown and prunes old session files;
- logical export/import and SQLite backup/restore;
- concurrent graph access and the existing observation/truth contracts.

## Consequences

SQLx is now evaluated against the behavior Asobi actually uses rather than against libSQL vector or replication capabilities. SQLite must be bundled with the default binary and compiled with FTS5. FTS5 remains ordinary SQLite SQL; semantic vector search is not a hidden fallback requirement.

This is a product-boundary decision for 0.6.0, not a claim that embeddings are never useful. A future document/vector feature can be introduced as an explicitly separate experiment once it has a daily workflow and bounded startup/maintenance costs.
